# Dose Optimization Frontend — Developer Documentation

## Project Overview

**Precision Rehabilitation Frontend** is a Next.js 15 web application that lets clinicians manage stroke-rehabilitation patients, upload observed outcome data, and request AI-generated dose-schedule predictions. It communicates with the FastAPI backend (`dose-optim-2026-dev`) for all model inference.

**Live URL:** https://precision-rehab-frontend-791208053302.us-west2.run.app  
**Backend URL:** https://precision-rehab-backend-791208053302.us-west2.run.app

## Key Concepts

### Patient Modes

Every patient has a `past: boolean` flag which determines which UI is shown on `/patient/[id]`.

**Past patients** (`past: true`)  
Historical trial subjects (source: `all_subjects_parameters.csv`). They have pre-fitted individual Bayesian parameters stored in the database. The UI shows the posterior-predictive trajectory alongside the actual observed scores. These are effectively read-only reference cases.

**Active patients** (`past: false`)  
New patients under current care. The clinician uploads weekly observations (dose + MAL/UEFM/WMFT scores). The UI allows:
1. **Manual Schedule** — clinician enters future doses; backend runs Stan HMC on the accumulated observations, then evaluates a posterior-predictive trajectory.
2. **Optimal Schedule (CEM)** — backend runs HMC to update the individual posterior, then runs the Cross-Entropy Method to find the dose schedule maximising expected outcome within the patient's budget.

### Outcome Metrics

| Metric | Scale | Description |
|---|---|---|
| MAL | 0–5 | Motor Activity Log (primary) |
| UEFM | 0–66 | Upper Extremity Fugl-Meyer |
| WMFT | 0–1 | Wolf Motor Function Test (normalised) |

The backend returns values normalised to [0,1]; the frontend scales them back for display.

### Adaptive Notebook Patients

Two special demo patients (IDs ending in `1` and `2`) replicate the Python notebook `adaptive_scheduling.ipynb`. When these patient IDs are detected in `api/manual-predict` or `api/optimize`, the frontend calls the backend's `/v1/hmc-predict` endpoint directly with hard-coded observation context from `src/lib/adaptiveNotebookPatients.ts` instead of the normal per-patient DB-backed flow.

## API Routes (Next.js server-side)

All API routes in `src/app/api/` run server-side only and forward calls to the FastAPI backend via `BACKEND_URL`.

| Route | Method | Purpose |
|---|---|---|
| `/api/patients` | GET | List all patients (fetches from backend, maps to frontend type) |
| `/api/patients` | POST | Create a new patient |
| `/api/patients/[id]` | GET | Get single patient with trajectory data |
| `/api/patients/[id]` | PUT | Update patient metadata |
| `/api/manual-predict` | POST | Run HMC + evaluate user's custom schedule |
| `/api/optimize` | POST | Run HMC + CEM to find optimal schedule |
| `/api/past-patient-plots/[subjectId]` | GET | Return pre-computed plot JSON from static files |
| `/api/model-params/[patientId]` | GET | Fetch fitted model parameters |
| `/api/prediction-results/[patientId]` | GET | Fetch cached prediction results |
| `/api/retrain/[patientId]` | POST | Trigger model retrain |

`src/lib/backend.ts` provides `backendGet`, `backendPost`, `backendPut` helpers and data-shape converters (`toFrontendPatient`, `enrichWithTrajectory`). **Never import `backend.ts` in a client component** — it uses server-only env vars.

## Running Locally

```bash
# Install dependencies
npm install

# Start dev server (with hot reload)
npm run dev
# → http://localhost:3000

# The backend must be running on port 8000.
# Either start it from dose-optim-2026-dev or use Docker Compose.
```

The frontend is also included in the root `docker-compose.yml`:

```bash
# From dose-optim-2026-dev/ run:
docker compose up
# Frontend at http://localhost:3001
# Backend at http://localhost:8000
```

## Building & Deploying to Cloud Run

See `../deployment.md` for the full step-by-step guide. Summary:

```bash
# From workspace root /Users/yezi/usc_ra/does-optim-dev

BACKEND_URL=https://precision-rehab-backend-791208053302.us-west2.run.app

# Build and push (must target linux/amd64 for Cloud Run)
docker buildx build \
  --platform linux/amd64 \
  --build-arg BACKEND_URL=$BACKEND_URL \
  -t us-west2-docker.pkg.dev/dose-optimization-app/dose-optim-images/frontend:latest \
  dose-optim-frontend \
  --push

# Deploy
gcloud run deploy precision-rehab-frontend \
  --image=us-west2-docker.pkg.dev/dose-optimization-app/dose-optim-images/frontend:latest \
  --region=us-west2 \
  --platform=managed \
  --port=3000 \
  --allow-unauthenticated \
  --set-env-vars="BACKEND_URL=$BACKEND_URL,NEXT_PUBLIC_APP_ENV=production"
```

## Data Flow for Active Patients

```
Clinician uploads CSV
        │
        ▼
UploadDataForm → POST /api/patients/[id] (save observations to backend DB)
        │
        ▼
NewPatientPage fetches updated patient data
        │
        ├──[Manual Schedule]──► ManualScheduleForm (enter future doses)
        │                             │
        │                             ▼
        │                       POST /api/manual-predict
        │                             │
        │                             ▼
        │                       backend /v1/hmc-predict or
        │                       /v1/patients/[id]/manual-predict
        │                             │
        │                             ▼
        │                       CurrentPredictChart (trajectory + CI band)
        │
        └──[CEM Optimize]────► POST /api/optimize
                                      │
                                      ▼
                                backend /v1/patients/[id]/optimize
                                (HMC posterior → CEM schedule → trajectory)
                                      │
                                      ▼
                                CurrentPredictChart (overlay CEM prediction)
```

## Static Past-Patient Plot Data

The directory `src/app/api/data/past-patient-plots/` contains one JSON file per trial subject (IDs 1–395). These are generated by the Python script `dose-optim-2026-dev/script/export_past_patient_plots.py`. When adding new past patients or regenerating trajectories, re-run that script and commit the updated JSON files.

## CSV Upload Format

The upload form expects a CSV with weekly rows. See `example_data/example.csv` for the exact column layout (week, dose_hours, mal_score, uefm_score, wmft_score).

## Key Files to Understand First

1. `src/types/patient.ts` — the central `Patient` and `PatientPred` interfaces
2. `src/lib/backend.ts` — how the frontend talks to the FastAPI backend
3. `src/app/patient/[id]/page.tsx` — routing logic that picks `PastPatientPage` vs `NewPatientPage`
4. `src/components/NewPatientPage.tsx` — the main active-patient interaction hub
5. `src/app/api/manual-predict/route.tsx` and `src/app/api/optimize/route.ts` — the two prediction API routes
